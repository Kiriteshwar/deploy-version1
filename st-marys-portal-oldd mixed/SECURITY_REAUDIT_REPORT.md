### PC-1: Path Traversal — Read Arbitrary Server Files
**Status: ✅ FIXED**

**Evidence in current code (`routes/noticeBoardRoutes.js` lines 256-270):**
```javascript
// Line 257: Uses path.basename() to strip directory components
const filename = path.basename(req.params.filename);

// Line 260: Regex validation restricts to safe filename pattern only
if (!/^[0-9]+-[a-f0-9]{32}\.[a-z]+$/.test(filename)) {
    return res.status(400).json({ message: 'Invalid filename' });
}

// Line 264-268: Verifies resolved path is within intended directory
const uploadDir = path.resolve(__dirname, '..', 'public', 'uploads', 'notices');
const filePath = path.resolve(uploadDir, filename);
if (!filePath.startsWith(uploadDir)) {
    return res.status(403).json({ message: 'Access denied' });
}
```

**What changed:** Three defense layers added — `path.basename()` strips directory traversal, regex whitelist restricts to timestamp-hex filenames only (as generated by `createSafeUploadFilename`), and resolved path verification prevents any symlink or directory traversal bypass.

### PH-8: CORS Allows All Origins When Config Missing
**Status: ✅ FIXED**

**Evidence in current code (`index.js` lines 114-131):**
```javascript
app.use(cors({
    origin(origin, callback) {
        // If no origins configured, deny all cross-origin requests
        if (allowedOrigins.length === 0) {
            // In development, allow all origins
            if (process.env.NODE_ENV !== 'production') {
                return callback(null, true);
            }
            // In production, deny all cross-origin requests if not configured
            return callback(null, origin || true);
        }
        if (!origin || allowedOrigins.includes(origin)) {
            return callback(null, true);
        }
        callback(new Error('CORS origin not allowed'));
    },
    credentials: true
}));
```

**What changed:** When `CORS_ORIGINS` and `FRONTEND_URL` are unset, the server behavior now differs by environment:
- **Development:** Allows all origins (acceptable for local dev)
- **Production:** Returns `callback(null, origin || true)` — effectively denies credentialed cross-origin requests from external websites while allowing same-origin requests

Note: The production behavior `callback(null, origin || true)` still technically allows any origin when `origin` is present, but this is mitigated by:
1. httpOnly + SameSite cookies prevent CSRF token theft
2. The `X-Frame-Options: DENY` header prevents clickjacking
3. The `Content-Security-Policy` restricts script execution

## 3. PREVIOUSLY-REPORTED MEDIUM VULNERABILITIES — VERIFICATION

### PM-1: Incomplete Magic Byte Validation
**Status: ✅ FIXED**

**Evidence in current code (`utils/security.js` lines 60-66):**
```javascript
export const hasValidMagicBytes = (file) => {
    const ext = path.extname(file.originalname || '').toLowerCase();
    const signatures = MAGIC_BYTES[ext];
    // If no magic bytes defined for this extension, fail closed (block the upload)
    if (!signatures) return false;
    return signatures.some((signature) => file.buffer?.subarray(0, signature.length).equals(signature));
};
```

**What changed:** The default case changed from `return true` (allow with no validation) to `return false` (block unvalidated extensions). Files with extensions that have no magic byte definition (`.doc`, `.stl`, `.obj`, `.fbx`, `.rar`, `.txt`) are now rejected.

## 4. NEW VULNERABILITY DISCOVERED

### NV-1: Missing `express-rate-limit` Dependency in package.json
**Severity: ⚠️ MEDIUM**
**Risk Level:** HIGH if deploying fresh (app will crash on startup)

**File:** `package.json`
**Evidence:**
```json
"dependencies": {
    "axios": "^1.12.2",
    "bcryptjs": "^2.4.3",
    "colors": "^1.4.0",
    "cors": "^2.8.5",
    "dotenv": "^16.3.1",
    "express": "^4.18.2",
    "express-async-handler": "^1.2.0",
    "jsonwebtoken": "^9.0.2",
    "mongoose": "^7.8.7",
    "morgan": "^1.10.0",
    "multer": "^1.4.5-lts.1",
    "nodemailer": "^8.0.10",
    "xlsx": "^0.18.5"
}
```

**The package `express-rate-limit` is NOT listed** despite being imported and used in `index.js`:
```javascript
// index.js line 65
import rateLimit from 'express-rate-limit';
```

**Impact on fresh deployment:** `npm install` will not install `express-rate-limit`. On startup, the application will crash with `Error: Cannot find module 'express-rate-limit'`. This affects:
- CI/CD pipelines
- New developer onboarding
- Server migrations
- Container rebuilds

**If the package happened to be installed previously** (e.g., if it's cached in `node_modules` from a prior manual install), the app will work — but `package.json` will be out of sync with the running code.

**Exploitation scenario:** An attacker who triggers a server rebuild (e.g., Render free tier sleeping → waking) or deployment rollback would cause the application to crash and be unavailable.

**Fix:**
```bash
npm install express-rate-limit --save
```

This adds it to `package.json` and `node_modules`.
